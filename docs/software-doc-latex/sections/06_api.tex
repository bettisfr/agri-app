\section{API Specification}

\subsection{Contract Strategy}
AgriApp uses \texttt{/api/v1} as the canonical integration layer for all active clients. The same API surface is consumed by:
\begin{itemize}
  \item RPi web UI (\texttt{/rpi/*}),
  \item Studio web UI (\texttt{/studio/*}),
  \item Android control app.
\end{itemize}

The practical objective is to keep behavior coherent while decoupling UI composition from backend logic.

\subsection{Export and Import Semantics}
Dataset export filenames include a server-side timestamp:
\begin{itemize}
  \item \texttt{agriapp\_dataset\_YYYYMMDD-HHMMSS.zip}
  \item \texttt{agriapp\_dataset\_visible\_YYYYMMDD-HHMMSS.zip}
\end{itemize}

The Studio import endpoint consumes ZIP files containing \texttt{images/}, \texttt{labels/}, \texttt{jsons/} (or \texttt{json/}), and \texttt{metadata/}. Import is best-effort with explicit report counters (\texttt{imported}, \texttt{skipped}, \texttt{errors}).

\subsection{Response Conventions}
Most routes return JSON and include an explicit status field. Binary responses are used for:
\begin{itemize}
  \item image downloads,
  \item ZIP exports,
  \item thumbnail streaming,
  \item ESP one-shot JPEG proxy output.
\end{itemize}

Pagination is server-side and stable (\texttt{page}, \texttt{page\_size}, \texttt{total\_items}, \texttt{total\_pages}), so web and Android can use identical navigation logic.

\subsection{Notes on Error Handling}
HTTP codes follow standard semantics:
\begin{itemize}
  \item \texttt{2xx}: success,
  \item \texttt{4xx}: request/state issues (missing params, invalid filenames, not found),
  \item \texttt{5xx}: runtime/backend execution failures.
\end{itemize}

Error payloads include concise diagnostic context to support field troubleshooting without immediate shell access.
