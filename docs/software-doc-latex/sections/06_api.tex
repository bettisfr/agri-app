The project adopts \texttt{/api/v1} as the primary and stable API namespace. All current and future client-side integrations should target this namespace first.

\subsection{Request/Response Semantics}
Most API endpoints exchange JSON payloads and return explicit status fields suitable for UI rendering and operational diagnostics. Download endpoints return binary streams (JPEG/ZIP). Pagination contracts are server-side and include canonical metadata (\texttt{page}, \texttt{page\_size}, \texttt{total\_items}, \texttt{total\_pages}) so that web and tablet clients can share a consistent navigation model. Thumbnail endpoints return lightweight JPEG streams backed by server-side cache to reduce list/grid payload cost.

Each image item exposed by \texttt{/api/v1/images} includes a \texttt{metadata} object with environmental and geospatial fields (\texttt{temperature}, \texttt{pressure}, \texttt{humidity}, \texttt{latitude}, \texttt{longitude}, \texttt{user\_comment}). During one-shot acquisition flows (\texttt{/api/v1/capture/oneshot} and \texttt{/api/v1/esp/capture}), the backend performs best-effort sampling of GPS and BME280 on Raspberry Pi and persists available values in dedicated sidecar files under \texttt{static/uploads/metadata}. This design avoids image recompression and keeps contextual information queryable in both browser and tablet galleries.

Annotation endpoints operate on image-scoped state and are responsible for maintaining consistency between JSON annotation files and YOLO TXT exports. Deletion endpoints enforce artifact cleanup behavior (image, label, JSON, metadata sidecar) to reduce orphaned files.

\subsection{Deprecation and Migration Note}
The project has completed a major cleanup step by removing key legacy aliases previously used for labels and single-image deletion. Ongoing maintenance should continue converging remaining historical routes toward the \texttt{/api/v1} namespace.
