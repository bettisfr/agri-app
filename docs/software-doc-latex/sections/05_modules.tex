\section{Software Modules}

\subsection{Capture Orchestration Module}
This module governs all image acquisition paths and normalizes them into a single storage lifecycle. It includes Raspberry Pi one-shot and loop capture, plus ESP32-CAM proxy capture. The operational goal is not only ``take a picture'' but ``produce a valid artifact that enters the gallery pipeline consistently''.

On Raspberry Pi, capture execution is delegated to \texttt{scripts/capture\_rpi.sh} and \texttt{client.py}, preserving camera tuning control while keeping API entry points clean. For ESP, the backend uses \texttt{scripts/capture\_esp.py} with controlled defaults (currently QXGA and high quality profile) and then stores the result locally with an \texttt{esp\_} filename prefix.

The module also owns loop lifecycle semantics (start, stop, status). This keeps unattended acquisition under API control and decouples day-to-day operations from manual shell sessions.

\subsection{Sensor Enrichment Module}
This module performs best-effort contextual sampling and metadata persistence. It integrates GPS and BME280 readings, merges them into a normalized metadata object, and writes sidecar files in \texttt{static/uploads/metadata}.

A key property is tolerance to partial availability. If one sensor is unavailable, available values are still persisted and capture remains successful. This design keeps the dataset usable even under non-ideal field conditions and avoids hard dependencies that would reduce acquisition continuity.

\subsection{Gallery Lifecycle Module}
The gallery module is responsible for listing, filtering, pagination, status inspection, open/delete operations, and dataset export orchestration. It is designed for scalability in practical terms: large galleries must remain usable without loading full-resolution assets in list views.

For this reason, thumbnail serving is part of the module contract, not a cosmetic optimization. Thumbnails are cached server-side and invalidated on deletion, allowing responsive client navigation while preserving full-resolution originals for detailed inspection.

\subsection{Labeling and Annotation Module}
The labeling module manages annotation state, editing semantics, and export compatibility. It supports box-level operations and class semantics while preserving dual persistence:
\begin{itemize}
  \item JSON as authoritative annotation state for UI reconstruction,
  \item YOLO TXT as training-oriented export.
\end{itemize}

Consistency between these artifacts is treated as a data integrity requirement. Save operations update the canonical state and maintain compatibility exports in the same transaction scope.

\subsection{Network and Discovery Module}
This module spans backend and Android responsibilities. On backend side, it exposes network mode APIs that call dedicated scripts to switch between Wi-Fi and AP operation. On Android side, it implements cache-first discovery and subnet-aware probing.

The module is intentionally context-sensitive: in known infrastructure Wi-Fi scenarios it avoids unnecessary ESP probing, while in AP rescue scenarios it supports full local control. This reduces latency and unnecessary network noise.

\subsection{System Control Module}
System control operations are exposed as dedicated high-impact actions: backend restart, Raspberry Pi reboot, and poweroff. This module is safety-oriented: actions are explicit, require deliberate UI confirmation, and return operational feedback suitable for logs.

Its value is practical: routine maintenance can be executed from tablet/web without relying on ad-hoc SSH access for every intervention.

\subsection{Client UI Modules}
Two client families consume the same backend semantics:
\begin{itemize}
  \item Web modules: gallery + labeler with browser-centric interactions,
  \item Android modules: control dashboard + paginated gallery + image detail dialog.
\end{itemize}

The project deliberately keeps business rules out of UI layers. Clients orchestrate and visualize; server APIs remain the source of truth.

\subsection{Module Cohesion}
The module graph is cohesive by design: capture produces images; sensor enrichment adds context; gallery and labeling transform raw artifacts into curated datasets; network/discovery keep the system reachable; system control preserves recoverability. This decomposition has proven practical during iterative refactoring because changes can be localized while preserving cross-module contracts.
